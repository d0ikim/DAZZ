# 01. Requirements — User Stories, NFRs, Edge Cases

> 이 문서는 **무엇을** 만들 것인지, **얼마나 잘** 만들어야 하는지, **무엇이 깨질 수 있는지**를 정의합니다.
> 모든 기능 구현은 여기 명시된 User Story 중 하나에 대응되어야 합니다.

---

## 1. Functional Requirements (User Stories)

### US-01. 음악적 계보 탐색 (입문자용)
> As a **재즈 입문자**,
> I want to **특정 아티스트의 프로필에서 그와 협업한 뮤지션들의 관계도를 시각적으로 확인하고 싶다**,
> so that **낯선 이름들 사이에서 길을 잃지 않고 '파도타기'하듯 내 취향의 아티스트를 자연스럽게 발견할 수 있다.**

- **연관 API**: `GET /api/v1/musicians/{musicianId}/insights?includeNetwork=true&depth=N`
- **핵심 기술 도전**: 그래프 깊이 탐색의 응답속도 (Redis 캐싱 필수)

### US-02. 심층 아카이빙 검색 (하드코어 팬용)
> As a **하드코어 재즈 팬**,
> I want to **단순 앨범 제목뿐만 아니라 특정 연주자가 참여한 세션 정보(Sideman 이력)까지 상세히 검색하고 싶다**,
> so that **좋아하는 연주자의 숨겨진 참여음반까지 놓치지 않고 디깅(Digging)하는 즐거움을 느낄 수 있다.**

- **연관 도메인**: `ALBUM_PARTICIPATION` (참여 관계 엔티티)
- **핵심 기술 도전**: 다중 JOIN 최적화 + 복합 인덱스 설계

### US-03. 셀프 브랜딩 포트폴리오 (뮤지션용)
> As a **현역 재즈 뮤지션**,
> I want to **나의 최신 공연이력, 앨범, SNS, 주로 활동하는 팀 정보를 직접 업데이트할 수 있는 전용 페이지를 갖고 싶다**,
> so that **나를 궁금해하는 대중이나 기획자에게 파편화된 정보가 아닌, 정돈된 '디지털 이력서'를 손쉽게 공유할 수 있다.**

- **연관 도메인**: `MUSICIAN ↔ USER` 1:1 Optional 관계
- **핵심 비즈니스 규칙**: 인증된 뮤지션만 본인 프로필 편집 가능

### US-04. 도슨트 큐레이션 가이드 (공용)
> As a **한국 재즈가 궁금한 사람**,
> I want to **'비 오는 날의 재즈피아노'처럼 무드와 장르별로 큐레이션된 아티스트 추천 가이드를 받고 싶다**,
> so that **전문지식이 없어도 도슨트의 설명을 듣듯 쉽고 재미있게 한국 재즈씬에 입문할 수 있다.**

- **연관 도메인**: `DocentNote`, `Curation`
- **장애 대응 시 핵심**: L3(Minimal) Fallback의 큐레이션 우회 (`/docs/08-fault-tolerance.md`)

## 2. Non-Functional Requirements (NFRs)

### NFR-01. 데이터 무결성 및 관계 일관성

| 항목 | 내용 |
| --- | --- |
| 지표 | 뮤지션-앨범-협업관계 데이터의 참조 무결성 **100%** 유지 |
| 근거 | '퍼스널 도슨트' 컨셉의 핵심은 정확한 정보 연결. 잘못된 연결 = 신뢰도 파괴 |
| 아키텍처 영향 | **RDBMS(MySQL) 선택**, ACID 트랜잭션 필수, 복합 인덱스 + 정규화 설계 |

### NFR-02. 탐색 반응속도 (Latency for Discovery)

| 항목 | 내용 |
| --- | --- |
| 지표 | '파도타기' 클릭 시 페이지전환/데이터로딩 **p99 0.5초 이내** |
| 근거 | Navigator 컨셉: 탐색 흐름이 끊기면 컨셉 자체가 무너짐 |
| 아키텍처 영향 | **Redis In-memory 캐시 필수**, SPA + API 최적화, Composite Index |

### NFR-03. 확장 가능성 및 가용성 (Scalability & Availability)

| 항목 | 내용 |
| --- | --- |
| 지표 | 피크 트래픽(평소 5배 이상) 대응, **99.9% 가용성** 보장 |
| 근거 | 공연 예매 오픈, 유명 연주자 이슈 등 트래픽 스파이크 대응 + 뮤지션의 포트폴리오 신뢰성 |
| 아키텍처 영향 | **Load Balancer + Stateless Scale-out**, AWS S3 + CDN, Kafka 비동기 |

## 3. Edge Cases (컨셉을 무너뜨릴 수 있는 케이스)

### EC-01. 허위/장난 뮤지션 프로필 선점
- **시나리오**: 검증되지 않은 사용자가 유명 뮤지션 이름으로 프로필을 만들거나, 허위 정보로 협업 데이터를 등록
- **위협**: '퍼스널 도슨트'의 신뢰도 파괴
- **Policy**:
  - 초기 가입 시 학력(서울예대 등) 및 활동 증빙 자료 업로드 요구
  - 관리자 승인 전까지 `Unverified` 마크 부여
  - 검증 등급별로 정보 노출 우선순위 차등

### EC-02. 동명이인 뮤지션 데이터 오염
- **시나리오**: '김재즈' 피아니스트와 '김재즈' 드러머의 데이터가 하나로 뒤섞여 관계도가 꼬임
- **위협**: 관계 그래프 무결성 붕괴
- **Policy**:
  - 모든 아티스트 객체는 **UUID 기반 고유 식별자**로 생성
  - 협업 데이터 입력 시 프로필 사진 포함된 '아티스트 선택 리스트'를 강제
  - 자유 텍스트 입력으로 인한 데이터 오염 원천 차단

### EC-03. 저작권 침해 및 외부 리소스 부하
- **시나리오**: 뮤지션이 저작권 걸린 고용량 영상/음원을 무분별하게 업로드
- **위협**: 법적 리스크 + 서버 부하 + 외부 API 응답 지연 전파
- **Policy**:
  - **직접 파일 업로드 금지**, YouTube/SoundCloud **임베드 방식만 허용**
  - 외부 API 응답 지연이 서비스 전체로 전파되지 않도록 **Circuit Breaker** 적용
  - 상세 설계 → `/docs/08-fault-tolerance.md`

## 4. 신뢰성 등급 (Trust Tier) 매트릭스

| Tier | 표시 | 권한 | 검증 방식 |
| --- | --- | --- | --- |
| **Verified Pro** | 파란 체크 | 본인 프로필 전체 편집, 라인업 등록 | 학력/활동 증빙 + 관리자 승인 |
| **Verified User** | 회색 체크 | 도슨트 노트 작성, 큐레이션 평가 | 이메일/실명 인증 |
| **Unverified** | 표시 없음 | 조회만, 댓글 제한 | 가입만 완료 |
| **Public Profile** | 'Unverified' 워터마크 | (시스템이 선등록한 뮤지션 프로필) | 본인이 나타나 인계 시 Verified로 전환 |

## 5. 우선순위 매트릭스 (MVP Scope)

| 기능 | 우선순위 | 근거 |
| --- | --- | --- |
| 공연 정보 조회 | **P0** | 사용자 유입 핵심 동인 |
| 뮤지션 프로필 + 관계도 | **P0** | 컨셉의 핵심 차별점 |
| 도슨트 노트 | **P0** | "Insight" 가치 전달 |
| 무드별 큐레이션 | **P1** | 입문자 진입 장벽 완화 |
| 세션 매칭 (비댓글형) | **P2** | 뮤지션 락인 효과 |
| 결제/예매 | **P3** | 수익화는 트래픽 확보 후 |
| 자유게시판 | **OUT** | 컨셉 분산 위험 |
