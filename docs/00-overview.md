# 00. Project Overview — DAZZ의 컨셉과 비즈니스 맥락

> 이 문서는 DAZZ의 **존재 이유(Why)**를 정의합니다.
> 모든 기능 설계, 기술 선택, UX 결정은 이 컨셉에 정렬되어야 합니다.
> 컨셉과 충돌하는 제안은 거절되어야 합니다.

---

## 1. The Concept — 3가지 관점의 자기 정의

### 1.1 3점 정리법 (A가 B하기 위해 C한다)

> **재즈 입문자(A)**가
> **복잡한 탐색 없이 한국재즈의 매력을 발견하고 향유(B)**할 수 있도록 돕는
> **올인원 재즈 아카이빙 플랫폼(C)**이 된다.

### 1.2 비교 강조법 (A가 아니라 B다)

> 이것은 **K재즈 백과사전(A)**이 **아니라**, **퍼스널 도슨트(B)**다.

이 구분이 의미하는 바:
- 백과사전은 **나열**한다 → DAZZ는 **연결**한다
- 백과사전은 **객관 정보**를 제공한다 → DAZZ는 **전공자의 해석(Insight)**을 제공한다
- 백과사전은 **검색**으로 끝난다 → DAZZ는 **다음 탐험지를 안내**한다

### 1.3 슬로건

> **"K-Jazz Insight Navigator"**

- **Information** ≠ **Insight**: 단순 정보가 아닌 통찰을 제공
- **Navigator**: 사용자를 다음 목적지로 안내하는 항해사

---

## 2. 해결하려는 문제 (Problem Statement)

### 2.1 재즈 입문자의 페인 포인트
- 낯선 한국 재즈 뮤지션의 이름 사이에서 길을 잃는다
- 어떤 앨범부터, 어떤 아티스트부터 듣기 시작해야 할지 모른다
- 정보는 흩어져 있고(SNS, 블로그, 공연 사이트), 맥락이 빠져있다

### 2.2 하드코어 팬의 페인 포인트
- 특정 연주자가 참여한 **세션 정보(Sideman 이력)**까지 디깅하기 어렵다
- 협업 관계, 사제 관계 등의 **계보**가 정리된 곳이 없다

### 2.3 현역 뮤지션의 페인 포인트
- 본인의 공연 이력, 앨범, 라인업 정보가 **파편화**되어 있다
- 기획자/팬에게 보여줄 **정돈된 디지털 이력서**가 없다

---

## 3. 핵심 가치 제안 (Value Proposition)

| 사용자 | 기존 경험 | DAZZ가 제공하는 가치 |
| --- | --- | --- |
| 재즈 입문자 | 검색 → 위키 → 막다른 길 | **'파도타기'식 관계 탐색** + 무드별 큐레이션 |
| 하드코어 팬 | 음반별 크레딧 수동 검색 | **Sideman 이력 통합 검색** + 협업 그래프 |
| 현역 뮤지션 | SNS 파편화 | **'디지털 이력서' 페이지** + 라인업 자동 연결 |

---

## 4. 핵심 기능 (한 줄 요약)

상세 User Story → `/docs/01-requirements.md`

1. **Interactive Relationship Graph**: 협업 횟수(가중치) 기반 동적 인맥지도
2. **Contextual Docent**: 협업 지점 클릭 시 시대적/음악적 배경 해설
3. **Real-time Collaboration Feed**: 새 협업 등록 시 Kafka 기반 실시간 그래프 업데이트
4. **셀프 브랜딩 포트폴리오**: 뮤지션 직접 관리 페이지
5. **무드별 큐레이션 가이드**: 비전문가 친화 도슨트 추천

---

## 5. 기술적 차별점 (왜 백엔드 엔지니어링이 본질인가)

- **고성능 조회**: 복잡한 그래프 연산을 Redis Cache-Aside 패턴으로 0.5s 이내 응답
- **데이터 정합성**: 동시 협업 데이터 등록 시 Redisson 분산락으로 정확한 가중치 계산 보장
- **시스템 안정성**: 알림/캐시 갱신 등 무거운 후속 작업은 Kafka로 비동기 처리하여 메인 서비스 보호

---

## 6. 컨셉 정합성 체크리스트 (의사결정 시 활용)

새 기능/설계 제안 시 다음 5가지를 자문하라:

- [ ] 이 기능은 **백과사전이 아닌 도슨트**의 역할을 강화하는가?
- [ ] 사용자에게 **다음 탐험지**를 제시하는가, 아니면 막다른 길에 세우는가?
- [ ] **전공자의 해석(Insight)**이 자연스럽게 녹아드는 구조인가?
- [ ] **연결의 품질**을 유지하는가, 데이터를 단순 나열하는가?
- [ ] 장애 발생 시에도 **도슨트로서의 안내 책임**을 포기하지 않는가?
  (→ 장애 대응에서도 큐레이션 우회 제공: `/docs/08-fault-tolerance.md`)

---

## 7. 안 만들 것 (Non-Goals)

명확히 컨셉에서 제외되는 것들:

- ❌ **익명 커뮤니티**: 뒷담화/비방 리스크. 실명/인증 기반만 허용
- ❌ **악기 장터**: 뮬(MULE)이 이미 점유. 컨셉 분산 방지
- ❌ **별점 리뷰**: 도슨트 컨셉과 상충 (정량 평가 ≠ 통찰)
- ❌ **자유게시판**: 초기 버전에선 커뮤니티 기능 최소화
- ❌ **글로벌 재즈 정보**: K-Jazz에 집중 (시장 차별화)
